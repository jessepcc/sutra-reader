# 經閣 / Sutra Reader — Spec (v0.3)

A mobile-first Progressive Web App for reading the CBETA Chinese Buddhist canon, sourced from [`cbeta-org/xml-p5`](https://github.com/cbeta-org/xml-p5). Visual language: Japanese zen / wabi-sabi. Open source. Local-only (no accounts).

---

## 1. Source & licensing

**Decisions.**

## 2. Scope

**In scope (v1)**
- Browse: 藏 (canon) → 冊 (volume) → 經 (text/juan)
- Read: rendered TEI with line/page references (`T01n0001_001`-style citations preserved)
- Per-text search (within the open scroll)
- 收藏 (Saved list) and per-position 標記 (Bookmarks) — see §6
- Recents (auto-tracked)
- Paper modes: 紙 / 墨 / 灰
- **Vertical RTL reading by default** (`writing-mode: vertical-rl`); horizontal LTR available as a per-device toggle for accessibility
- Offline reading of any text the user has opened
- Background sync against upstream repo
- **UI chrome: 繁體中文 only in v1.** English/Japanese chrome deferred to v2.

**Out of scope (v1)** — held firm to prevent scope creep:
- User accounts / cloud sync of any kind
- Annotations, highlights, or notes
- Full-canon FTS (corpus too large to index client-side)
- Cross-reference / dictionary lookups
- Audio recitation, translations, sharing of personal annotations

## 3. Data architecture

### 3.1 Catalog
- Bundled in the app build: a frozen copy of `canons.json` plus a generated `catalog.json` listing every text file with `{canon, volume, id, path, sha, bytes}`. Excludes gated canons (§1).
- App ships with **catalog only**, not text bodies. Catalog target < 2 MB gzipped.

### 3.2 Text fetch
- On first open of a text, fetch from jsDelivr pinned to commit SHA:
  ```
  https://cdn.jsdelivr.net/gh/cbeta-org/xml-p5@<sha>/<path>
  ```
  Preferred over `raw.githubusercontent.com`: edge cache, no per-IP rate limits, supports commit-pinned URLs.
- Store raw XML + rendered HTML in IndexedDB, keyed by `{textId, sha}`.

### 3.3 Sync to upstream
- A **manifest** file is regenerated by CI on a 6-hour schedule:
  ```json
  { "generatedAt": "...", "upstreamSha": "...", "files": [{ "path": "...", "sha": "..." }, ...] }
  ```
  Hosted at a stable URL the PWA polls.
- On launch (and via Background Sync API where supported), the PWA:
  1. Fetches the latest manifest.
  2. Diffs SHAs against locally cached texts.
  3. Marks changed texts as **stale** (subtle badge) and re-fetches lazily on next open, or eagerly if the user enabled "auto-update".
- Settings → "檢查更新" runs the same check on demand.
- Code release required for *new* texts to appear in browse (catalog ships in bundle). v2 may load catalog from the manifest server.

### 3.4 TEI rendering
- v1 supports the focused subset of TEI P5 as used by CBETA:
  - `<div type="juan">`, `<head>`, `<p>`, `<lb n="…"/>`, `<pb n="…"/>`, `<lg>`/`<l>`, `<note>`, `<g ref="…"/>` (gaiji), `<choice><orig/><reg/></choice>`, `<app><lem/><rdg/></app>`.
  - Gaiji: resolve via the CBETA gaiji table (bundled JSON); render Unicode where available, SVG fallback otherwise. Long-press shows the original glyph reference.
- Transform runs in a Web Worker; output cached as HTML fragments.
- Unknown elements are logged to a "rendering issues" diagnostic panel, never silently dropped.

## 5. Local storage & data ownership

**Principle: everything is on-device. No accounts, no servers, no telemetry by default.**

### 5.1 Storage map (IndexedDB via `idb`)

| Store | Contents | Bounded by |
|---|---|---|
| `texts` | `{textId, sha, xml, htmlFragments[]}` | LRU cache, default 200 MB, user-configurable |
| `saved` | `[{textId, savedAt, note?}]` | unbounded (small) |
| `bookmarks` | `[{textId, lb, label, createdAt}]` | unbounded (small) |
| `recents` | `[{textId, openedAt, lastLb}]` | last 50 |
| `settings` | `{direction, paperMode, fontScale, autoUpdate, …}` | single object |

`localStorage` is used only for tiny ephemeral keys (last route, theme bootstrap) to avoid FOUC; everything durable is IndexedDB.

### 5.2 Save function (收藏)

- **Distinct from Bookmarks.** Saved (收藏) marks the whole text for quick access from the home/Saved screen. Bookmarks (標記) mark specific positions within a text.
- Brush-styled star/enso icon in the Reader header toggles saved state.
- Saved list appears as a top-level nav surface with the most recent saves first; long-press to reorder or remove.
- Saved texts are opportunistically pre-cached when on Wi-Fi (so they remain available offline).

### 5.3 Backup / portability (compensates for no account)

- Settings → "匯出資料 Export" produces a single JSON file:
  ```json
  { "version": 1, "savedAt": "...", "saved": [...], "bookmarks": [...], "settings": {...} }
  ```
- Settings → "匯入資料 Import" merges from a chosen JSON file (conflicts resolved by latest timestamp).
- Settings → "清除資料 Clear" shows what's stored and lets the user wipe selectively (cache only / saves only / everything).

### 5.4 Privacy stance (write into About + README)

- No analytics, no third-party scripts, no fingerprinting.
- Only outbound network calls: jsDelivr for text fetch, the manifest endpoint for sync. Both documented in About.

## 6. Visual design — 侘寂 wabi-sabi

### Palette (strict three-stop greyscale; no chroma)

| Token | Light | Dark |
|---|---|---|
| `--ink` (sumi 墨) | `#1a1a1a` | `#e8e6e1` |
| `--paper` (washi 和紙) | `#f4f1ea` | `#15140f` |
| `--mist` (kasumi 霞) | `#9b9690` | `#6b665e` |
| `--rule` (hairline) | `#c9c4ba` | `#2a2823` |

Off-white paper, not pure white — pure white reads sterile.

### Typography
- Headings / brand: a CJK brush face — `"Klee One"`, `"Yuji Mai"`, or `"Shippori Mincho B1"` from Google Fonts, subset to the glyphs used in chrome.
- **Body reading: `"Noto Serif TC"` (Traditional Chinese)**. Fallback chain `"Noto Serif TC", "Source Han Serif TC", serif`. Content is Classical Chinese, not Japanese — the design *frame* is Japanese-inspired, the *reading typeface* must be Chinese-first or rare canonical glyphs render as tofu.
- UI chrome: `"Inter"` for small Latin labels; numerals tabular.

### Spacing & composition
- Fibonacci scale: 8 / 13 / 21 / 34 px. Feels organic, not gridded.
- Asymmetric, off-center layouts. Reader column offset from page center.
- Hairline rules (`1px solid var(--rule)`) instead of boxes / cards.
- No drop shadows. Max corner radius 2 px anywhere.

### Brush / sumi-e accents (used sparingly — wabi-sabi is restraint)
- App mark: a single hand-painted enso (円相) SVG.
- Section dividers: ink-wash horizontal smear SVG, randomly mirrored per render so no two pages divide identically.
- Reading-progress indicator: thin ink-stroke that grows along the leading edge as you scroll; ragged, brush-textured.
- Route transitions: subtle ink-bleed wipe (~180 ms, `cubic-bezier(.2,.8,.2,1)`).
- Subtle paper-grain SVG noise on `--paper` (~3% alpha). Disabled on `prefers-reduced-motion` and low-end devices.

### Iconography
Thin (1.25 px) hand-drawn strokes. No filled icons. No Material/HIG defaults.

## 7. Mobile-first behavior

- Single-column reading column max 38em; tap zones ≥ 44 px.
- **Vertical RTL is the default reader mode** (`writing-mode: vertical-rl`); horizontal LTR available as per-device toggle.
- Pinch-zoom and dynamic font-scale both honored.
- Pull-to-refresh on Browse re-checks the manifest; **disabled inside Reader** to avoid accidental scroll triggers.
- Safe-area insets respected (notch / home indicator).
- `prefers-color-scheme` and `prefers-reduced-motion` respected.

## 8. PWA / technical

**Stack (recommended, not prescriptive)**
- SvelteKit or Next.js (static export). Trivial to host on GH Pages or Cloudflare Pages.
- Service worker via Workbox:
  - App shell: cache-first.
  - Catalog & manifest: stale-while-revalidate.
  - Text XML on jsDelivr: cache-first, invalidated on SHA change.
- IndexedDB via `idb`. LRU eviction at user-configurable cap (default 200 MB).
- XML → HTML transform in a Web Worker (`fast-xml-parser` or `sax-wasm`).
- Routing: deep-linkable (`/read/<canonId>/<textId>?lb=001a05`), no hash.

**Web app manifest (`manifest.webmanifest`)**
- `display: "standalone"`, `orientation: "any"`
- `theme_color: "#1a1a1a"`, `background_color: "#f4f1ea"`
- Icons: enso brush mark at 192 / 512 / maskable
- `share_target` so users can share a citation into the app

**Performance budget**
- First contentful paint < 1.5 s on mid-tier mobile, 4G
- Reader open → first painted line < 800 ms for cached texts; < 3 s for cold fetch of a typical (~200 KB) sutra
- Long sutras (e.g., `T01n0001` at 3 MB) stream in juan-by-juan
- Total JS < 150 KB gzipped for app shell

**Accessibility**
- Dyslexia-friendly font alternative in Reader
- Min contrast 4.5:1 in all paper modes
- Keyboard navigable (arrow keys for browse, J/K paging in Reader)
- Brush ornaments decorative (`aria-hidden`)

## 9. Hosting

**Primary: Cloudflare Pages.**
- Unmetered bandwidth (GH Pages soft-caps at 100 GB/mo)
- Edge cache in Asia-Pacific is materially faster for likely audience
- Per-PR preview deploys
- Optional Worker if the manifest ever needs to be dynamic

**Fallback: GitHub Pages.** Pure static build, no platform-specific build steps in app repo, so the deploy target is interchangeable.

**Public URL.** Use whichever default URL the chosen platform provides:
- GH Pages → `https://<user>.github.io/sutra-reader/`
- CF Pages → `https://sutra-reader.pages.dev` (or similar)

No custom domain in v1.

## 10. Repo layout (open source)

Two public repos under one GitHub namespace:

| Repo | Purpose | License |
|---|---|---|
| `sutra-reader` | PWA application code (this directory) | **MIT** |
| `sutra-manifest` | CI-generated catalog + sync manifest | MIT (code) + CC BY-NC-SA 3.0 TW (data) |

### Required files at repo root
- `LICENSE` — MIT text
- `NOTICE.md` — full CBETA attribution + exceptions list, verbatim from §12
- `README.md` — what it is, screenshot, dev/deploy instructions, **prominent non-commercial notice**
- `CONTRIBUTING.md` — DCO sign-off; no PRs that add ads, tracking, or telemetry without explicit opt-in
- `CODE_OF_CONDUCT.md` — Contributor Covenant
- `SECURITY.md` — private disclosure address
- `SPEC.md` — this document

### CI / deploy
- `ci.yml` — typecheck, lint, build, Lighthouse PWA audit (fail < 90 on PWA / Performance)
- `deploy.yml` — on push to `main`, deploy to chosen platform
- `sync.yml` (in `sutra-manifest`) — `schedule: '0 */6 * * *'`, regenerates catalog + manifest, commits if changed

### Hygiene constraints
- No analytics phoning home by default. Any future telemetry is opt-in, self-hostable, no third-party trackers.
- No bundled text content in the app repo — keeps repo light and keeps content license cleanly separated.
- Reproducible builds: pinned Node version, lockfile committed.
